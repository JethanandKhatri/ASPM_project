Introduction

What Is This System?

Read this section before the demo — all three members must understand this.

ASPM CASE Tool is a complete web-based project management system designed for Agile software teams. Instead of using separate tools (Jira for tasks, Confluence for docs, Excel for estimations), our system puts everything in one place — with different views tailored to each team role.

Core Concept: Every person in a software team has a different job. A manager needs budgets and risks. A developer needs their tasks. A Scrum Master needs to run sprints. Our tool gives each person exactly what they need — nothing more, nothing less.

Key selling point for Professor: Every role feeds data to the next role automatically. PM's risks become SM's impediments. PM's estimation becomes SM's capacity reference. Developer's standup appears in SM's view. This is a closed-loop information system — not just isolated screens.

Professor Q&A — Expected Questions & Answers

Study these before the demo. The question column notes which member should answer each one.

Q Why did you use Supabase instead of Firebase or a custom backend?
Supabase provides built-in authentication with role-based user creation via an admin API. This is essential for our Admin-creates-accounts flow — we can create real Supabase Auth users programmatically from the frontend. Firebase Auth doesn't expose the same admin-level user creation in the client SDK. Supabase also gives us a real PostgreSQL database instead of a NoSQL store, which suits our relational data model (projects → tasks → sprints).
📌 Member 1 answers
Q Explain COCOMO. How did you implement it?
COCOMO stands for Constructive Cost Model, developed by Barry Boehm. The formula is: Effort = a × (KLOC)^b, where KLOC is thousands of lines of code. The constants a and b vary by project type — Organic projects use a=2.4, b=1.05; Semi-detached use a=3.0, b=1.12; Embedded use a=3.6, b=1.20. Duration is then calculated as: Duration = 2.5 × (Effort)^0.38. We implemented this formula directly in the TechniqueSelector component with the PM entering KLOC and selecting project type.
📌 Member 1 answers
Q What is WIP Limit and why does it matter in Scrum?
WIP stands for Work In Progress. The WIP Limit is a Lean/Kanban principle that restricts how many tasks can be in the "In Progress" column simultaneously. The research behind it — from the Theory of Constraints — shows that excessive multitasking reduces productivity due to context-switching overhead. Limiting WIP forces the team to finish tasks before starting new ones, improving flow and reducing lead time. In our system, when the WIP limit is exceeded, the In Progress column turns red and shows a warning, alerting the SM to intervene.
📌 Member 2 answers
Q How does the PM's risk register connect to the Scrum Master?
Both the PM's Risk Register and the SM's Impediments tab read from the same Supabase risks table. When the PM adds a risk with status "Open" or "In Progress", it automatically appears in the SM's Impediments list labeled as "From risks". The SM does not need to be told about the risk manually — the system surfaces it. When the PM resolves the risk (marks it as "Resolved"), it disappears from the SM's impediments list. This is real-time cross-role integration through a shared database.
📌 Member 2 answers
Q Why can't the developer see all sprint tasks? Is that a missing feature?
This is an intentional design decision based on Scrum principles. In professional Scrum, each team member is responsible for their own tasks. Showing a developer all 50 tasks from the entire team creates unnecessary cognitive load and distracts from their focus work. The Scrum Master has full visibility because it is their responsibility to monitor the whole team. The developer has filtered visibility because it is their responsibility to deliver their specific tasks. This is the "focus" principle from the Scrum Values.
📌 Member 3 answers
Q What happens to tasks that are not finished when a sprint ends?
When the SM completes a sprint, the system separates completed tasks (status = Done) from incomplete ones. Completed tasks remain in the sprint record. Incomplete tasks are removed from the sprint's task list and return to the backlog automatically — they appear as unassigned tasks available for re-prioritization in the next sprint planning session. The sprint record also saves the names of dropped tasks as a log, which is visible on the sprint card so the PM can see exactly what was incomplete when reviewing sprint history.
📌 Member 2 answers
Q How does Fuzzy Logic estimation work in your system?
Fuzzy Logic handles uncertainty in estimation by using linguistic variables instead of precise numbers. Instead of saying "this will take exactly 120 hours," the team says "the project size is Medium and the complexity is High." Our system maps these linguistic inputs to fuzzy membership functions, applies fuzzy inference rules (e.g., IF size is Medium AND complexity is High THEN effort is High), and uses defuzzification to produce a crisp number output — effort in person-days. This is particularly useful in early project stages when precise requirements are not yet known.
📌 Member 1 answers
Q How is the capacity recommendation calculated?
The system calculates recommended story points using two inputs: (1) team velocity from completed sprints — the average story points or tasks completed per sprint, (2) total available person-days for the upcoming sprint from the capacity table. It then calculates velocity per person-day from historical data and multiplies it by the available person-days to produce a recommended commitment. For example: if past sprints averaged 40 SP with 20 available person-days each, velocity is 2 SP/person-day. If this sprint has 15 available person-days, the recommendation is 30 SP.
📌 Member 2 answers
Q What is a Sprint Planning Artifact and where is it in your system?
A Sprint Planning Artifact is a formal document that captures the sprint commitment — what stories are being delivered, how many story points are committed, team capacity, and progress tracking. In our system, it appears automatically in the Sprints tab when a sprint is active. It shows: committed user stories, story points per story, number of tasks per story, tasks completed, and a progress bar per story. This is generated dynamically from the actual sprint data — no manual document creation needed.
📌 Member 2 answers
Q What are the limitations of your system?
Three main limitations: (1) Database-level security: We use the Supabase admin client for all queries, which bypasses Row Level Security policies. In production, we would add per-user RLS policies. For an academic shared workspace this is acceptable. (2) Retrospective persistence: The retrospective is stored in browser localStorage, meaning it is device-specific. In a production system, retrospective data would be stored in Supabase. (3) Velocity chart: We removed the velocity chart to simplify the SM portal — a production ASPM tool would include this as a standard Scrum metric for sprint-over-sprint improvement tracking.
